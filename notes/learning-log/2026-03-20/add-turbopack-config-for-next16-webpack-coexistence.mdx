## Context
After upgrading to Next.js 16, build failed because Turbopack is enabled by default and the project still uses a custom `webpack` config.

## Goal
Allow Next.js 16 build to proceed without removing existing webpack plugin behavior.

## Files Changed
- `next.config.mjs`

## Implementation Notes (key logic explained)
Added `turbopack: {}` to Next config. This explicitly acknowledges Turbopack configuration presence while retaining the existing custom webpack plugin block.

## Key Decisions (and alternatives considered)
- Decision: add minimal `turbopack` config entry.
- Alternative: remove custom webpack config entirely. Rejected because Velite integration depends on it.
- Alternative: force `--webpack` flags in scripts. Deferred for now to keep script changes minimal.

## Assumptions
- Existing webpack path should remain functional for current build workflow.

## Failure Modes
- If Turbopack and custom webpack behavior diverge further, scripts may need explicit `--webpack`.

## How to Test (commands + steps)
1. Run `npm run build`.
2. Confirm Next 16 no longer errors on missing Turbopack config.

## Next Steps
- If build still fails, set `next build --webpack` and `next dev --webpack` explicitly.
