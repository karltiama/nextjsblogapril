## Context

The NBA project implementation content is stored as a long multiline
string in `projectsDataV2.ts`. On the project detail page, it rendered
inside a single `<p>`, making the content dense and hard to scan.

## Goal

- **Goal:** Improve readability of the Implementation section
  formatting.
- **Inputs:** Existing implementation text containing paragraph breaks
  via double newlines.
- **Output/Done:** Implementation now renders as multiple paragraphs.

## Files Changed

- `app/projects/[projectName]/page.tsx`

## Implementation Notes (key logic explained)

- Added `implementationParagraphs`:
  - splits implementation text on `\n\n`
  - trims each chunk
  - filters empty chunks
- Updated Implementation render block:
  - if parsed paragraphs exist, map to multiple `<p>` tags with
    vertical spacing
  - fallback to a single `<p>` for compatibility

## Key Decisions (and alternatives considered)

- **Split by double newline:** simple, explicit formatting contract in
  content.
- **Alternative:** store implementation as `string[]` in data model;
  cleaner type but larger migration scope.

## Assumptions

- Content authors use blank lines to indicate paragraph boundaries.

## Failure Modes

- If content has no blank lines, it still renders as one paragraph
  (fallback).
- If future content needs subheadings/list styling, a richer data
  schema may be preferable.

## How to Test (commands + steps)

1. `npm run dev`
2. Open `/projects/nba-analytics-app`.
3. Confirm Implementation renders as separated paragraphs with
   spacing.
