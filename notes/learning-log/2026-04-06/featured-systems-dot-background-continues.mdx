## Context

The home (and terrain) page used a dotted radial overlay only on the “Writing & Thinking” block. “Featured Systems” sat below with the same base `#0a0a0f` fill from the root wrapper but **no** dot layer, so the pattern visibly stopped at the section boundary.

## Goal

Let the dot grid read as one continuous background through Writing and Featured Systems.

## Files Changed

- `app/page.tsx`
- `app/terrain/page.tsx`

## Implementation Notes (key logic explained)

- Introduced a **single** full-width wrapper (`relative isolate overflow-hidden bg-[#0a0a0f]`) that contains both sections.
- Moved the absolute dot layer to that wrapper with `inset-0` so it spans the **combined height** of blog + projects. Content sections use `relative z-[3]` so they stack above the pattern.
- Removed the duplicate dot layer from the inner blog `<section>` (it only covered the first block before).

## Key Decisions (and alternatives considered)

- **One wrapper vs. duplicate dots on projects:** Duplicating the overlay on the second section would restart the pattern at that section’s top-left; a shared parent keeps phase alignment across the fold between sections.
- **Fixed attachment:** Could use `background-attachment: fixed` for viewport-locked dots; not needed here because one parent already unifies both sections.

## Assumptions

- Terrain page should match home for the same visual continuity.

## Failure Modes

- If future content is inserted **between** the two sections inside the wrapper, it will also get dots (usually desired). Content **outside** the wrapper (e.g. below projects) still has no dots unless added.

## How to Test (commands + steps)

1. `npm run build`
2. Open `/` and scroll from “Writing & Thinking” into “Featured Systems”; confirm the dot texture does not reset at the heading.
3. Optionally open `/terrain` and repeat.

## Next Steps

- If `/projects` or other routes should match, reuse this wrapper pattern or extract a small layout component.
