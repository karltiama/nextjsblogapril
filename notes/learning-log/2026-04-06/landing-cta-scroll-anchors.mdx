## Context
The landing page hero had two primary CTAs ("Read the Blog" and "View Projects") that navigated to separate routes instead of scrolling to the corresponding sections on the same page.

## Goal
Make both hero CTA buttons scroll to the blog and projects sections on the landing page, with reliable positioning across responsive breakpoints.

## Files Changed
- `app/page.tsx`

## Implementation Notes (key logic explained)
- Updated hero CTA link targets from route paths to hash anchors:
  - `/blog` -> `/#blog-section`
  - `/projects` -> `/#projects-section`
- Added section IDs on landing sections:
  - `id="blog-section"` on the writing/blog preview section
  - `id="projects-section"` on the projects section
- Added `scroll-mt-24` to both sections so anchored scrolling offsets for sticky header space and lands correctly at common viewport sizes.

## Key Decisions (and alternatives considered)
- Used native hash anchor navigation for simplicity and zero extra client logic.
- Chose `scroll-mt-24` to handle header overlap in a single utility class.
- Alternative considered: custom JS `scrollIntoView` handlers with dynamic offsets; rejected as unnecessary complexity for this use case.

## Assumptions
- The request refers specifically to hero CTA behavior on the landing page.
- Existing site header can overlap anchored sections without scroll margin.

## Failure Modes
- If header height changes significantly, `scroll-mt-24` may need tuning.
- If future refactors remove or rename section IDs, anchor links will stop targeting correctly.

## How to Test (commands + steps)
1. Start dev server:
   - `npm run dev`
2. Open the landing page (`/`).
3. Click "Read the Blog" and verify the page scrolls to the "Writing & Thinking" section.
4. Click "View Projects" and verify the page scrolls to the "Featured Systems" section.
5. Repeat tests on mobile width, tablet width, and desktop width.
