## Context
`npm run build` failed after terrain integration due to missing type declarations for `three` and `three/examples/...`, plus downstream prop typing errors in `<meshStandardMaterial />`.

## Goal
Restore successful TypeScript type-checking for the terrain hero without changing runtime behavior.

## Files Changed
- `package.json`
- `package-lock.json`

## Implementation Notes (key logic explained)
Installed `@types/three` as a dev dependency. React Three Fiber's JSX material prop typings depend on Three.js types, so missing declarations caused both module type errors and invalid material prop typing.

## Key Decisions (and alternatives considered)
- Decision: add `@types/three`.
- Alternative: add local `declare module` shims for `three` and `ImprovedNoise`. Rejected because it would degrade type safety and hide real typing issues.

## Assumptions
- Project uses TypeScript strict enough for missing declaration files to fail build.

## Failure Modes
- If major versions of `three` and `@types/three` drift too far apart, new type mismatch errors can appear.

## How to Test (commands + steps)
1. Run `npm run build`.
2. Confirm no type errors in `components/terrain-hero.tsx`.
3. Run `npm run dev` and verify terrain still renders.

## Next Steps
- Keep `three` and `@types/three` aligned during dependency updates.
