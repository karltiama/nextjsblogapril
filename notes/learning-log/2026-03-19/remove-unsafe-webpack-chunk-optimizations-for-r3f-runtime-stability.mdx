## Context
Runtime errors (`ReactCurrentOwner`) and repeated missing `.next` chunk files indicated unstable bundling behavior in development after adding React Three Fiber.

## Goal
Stabilize module bundling/runtime by reverting risky Next/Webpack experimental optimizations.

## Files Changed
- `next.config.mjs`

## Implementation Notes (key logic explained)
Removed:
- `experimental.optimizePackageImports`
- custom client `splitChunks` cache group overrides

Kept Velite plugin integration unchanged. This returns bundling behavior closer to Next.js defaults, which is generally safer for packages with complex runtime internals like `@react-three/fiber`.

## Key Decisions (and alternatives considered)
- Decision: rollback custom bundling tweaks first.
- Alternative: downgrade Fiber/Three immediately. Not chosen first because chunk-level instability was already visible in logs and can independently cause the observed runtime issues.

## Assumptions
- Existing custom chunk optimization settings were introduced to mitigate unrelated chunk issues but now conflict with current dependency/runtime behavior.

## Failure Modes
- If error persists, dependency-level mismatch still exists and package pinning is the next step.
- Build output characteristics (chunk sizes/names) may change after reverting custom split config.

## How to Test (commands + steps)
1. Stop all running `next dev` processes.
2. Run `npm run dev:clean`.
3. Open `/` and verify terrain renders without `ReactCurrentOwner` errors.
4. Navigate `/about`, `/projects`, `/blog` and verify no chunk missing errors.

## Next Steps
- If still failing, pin `@react-three/fiber` and `three` to a known stable pair for React 18 and retest.
