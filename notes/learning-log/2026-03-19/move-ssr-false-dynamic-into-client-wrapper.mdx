## Context
Build still failed during prerender with `ReactCurrentBatchConfig` even after moving away from `ssr: false` in `app/page.tsx`.

## Goal
Ensure React Three Fiber code is loaded only on the client at runtime.

## Files Changed
- `components/home-terrain-hero.tsx`

## Implementation Notes (key logic explained)
Updated the client wrapper to use `next/dynamic` with `ssr: false` when importing `terrain-hero`. This keeps the server page clean while preventing server/prerender evaluation of Fiber-heavy code paths.

## Key Decisions (and alternatives considered)
- Decision: place `dynamic(..., { ssr: false })` inside a Client Component wrapper.
- Alternative: convert `app/page.tsx` to client. Rejected due to larger scope and server-rendering trade-offs.
- Alternative: remove terrain from homepage at build time. Rejected because feature should remain present.

## Assumptions
- Next.js allows `ssr: false` inside Client Components (it disallows it in Server Components).

## Failure Modes
- If wrapper is imported incorrectly, terrain will not render.
- If a different React/Fiber compatibility issue exists, runtime client errors may still occur.

## How to Test (commands + steps)
1. Run `npm run build`.
2. Confirm no prerender error for `/`.
3. Run `npm run dev`.
4. Open `/` and verify terrain hero renders.

## Next Steps
- If errors persist, pin a known-compatible `three` version for Fiber v8 (e.g. `three@0.15x`) and retest.
