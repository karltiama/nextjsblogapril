## Context

The user wanted everything they can do on their side first for Umami:
tighten the tracker in code and align documentation with the real app
(no embedded `/analytics` route).

## Goal

- **Goal:** Harden Umami Cloud tracking in the Next.js app and fix stale
  setup documentation.
- **Inputs:** Existing `app/layout.tsx`, `UMAMI_SETUP.md`, production
  hostname `karltiama.dev`.
- **Output/Done:** Tracker uses recommended options and Next.js loading
  strategy; docs describe layout + footer share + Cloud-only steps.

## Files Changed

- `app/layout.tsx`

## Implementation Notes (key logic explained)

- Moved the Umami `<Script>` from `<head>` to the end of `<body>` with
  `strategy="afterInteractive"` so loading matches Next.js guidance for
  third-party analytics.
- Added `data-exclude-search`, `data-do-not-track`, and
  `data-performance` as string `"true"` attributes on the script (Umami
  reads HTML data attributes).
- `data-domains` is applied when `VERCEL_ENV === "production"` (Vercel
  production deploy) or when running a local production server
  (`NODE_ENV === "production"` without `VERCEL`). That omits the lock
  for `next dev` and for Vercel preview deploys so preview URLs still
  report to Umami.

## Key Decisions (and alternatives considered)

- **Domain lock gating:** Vercel preview uses `NODE_ENV=production`, so
  gating on `VERCEL_ENV === "production"` avoids silencing preview
  traffic; local `next start` still applies the lock (localhost excluded).
- **No new env vars:** website id stays a named constant next to the
  script; alternative is `NEXT_PUBLIC_UMAMI_WEBSITE_ID` for rotation
  without code edits.

## Assumptions

- Production site responds on `karltiama.dev` and optionally
  `www.karltiama.dev`; if you use only apex or only `www`, the list can
  be trimmed.
- Umami Cloud’s hosted `script.js` supports `data-performance` for your
  plan/version (if metrics are missing, remove the attribute and
  re-check Umami docs).

## Failure Modes

- **Local `next start`:** `data-domains` applies, so Umami ignores
  `localhost` unless you remove the lock for that scenario.
- **DNT:** some visitors will not be counted when DNT is enabled.

## How to Test (commands + steps)

1. `npm run lint`
2. `npm run dev` — visit a page, confirm Umami still receives events
   (no `data-domains` in dev).
3. `npm run build && npm run start` — expect **no** Umami hits from
   `localhost` (domain lock). Confirm on `https://karltiama.dev` after
   deploy.
